# 開発ワークフロー

## ドキュメントファースト

実装を始める前に、必ず以下を確認する:

1. `docs/spec/` に該当する accepted な仕様書があるか確認
2. `docs/adr/` に関連する accepted な意思決定記録があるか確認
3. なければ、先にドキュメントを作成し承認を得る

## 開発フロー全体像

```text
要求整理 → 仕様策定 → 設計 → 実装 → レビュー → リリース
```

### 1. 要求整理

- コアバリューの定義（プロダクトが提供する最大の価値）
- 競合調査・市場調査（deep-research等を活用）
- ターゲット層の決定
- 優先度付け → MVP決定
- ブランドカラー・世界観の構築
- 成果物: `docs/research/` および `docs/adr/`

### 2. 仕様策定

- wip で spec ドキュメントを作成
- ロードマップ軸で要件を分割（MVP / Phase 1 / Phase 2...）
- デザイントークンで表現を制約（`docs/design-tokens/` 参照）
- モジュール独立性を意識（クリーンアーキテクチャ等）
- 固まったら accepted に変更
- 成果物: `docs/spec/`

### 3. 設計

- ADR に設計判断を記録
- 技術選定の根拠を残す
- 問題発生時に判断根拠を持って再評価できるようにする
- 成果物: `docs/adr/`

### 4. 実装

- accepted な spec に基づいて実装
- TDD を推奨: テスト → 実装 → 検証 → リファクタ
- 小さな差分で段階的に進める

### 5. レビュー

- 過去の指摘パターンからレビュー指針を蓄積
- コンパイルエラーとCIで自動修正を先に実施
- 人間のレビューは設計判断とビジネスロジックに集中

### 6. リリース

- リリースチェックリストに従う
- リリース後のフィードバックを次の要求整理に反映

## フィードバックループ

実装品質を高めるための自動フィードバック:

- **コンパイルエラー**: エラー出力をAIに渡して自動修正
- **テスト失敗**: 失敗内容をAIに渡して修正案を生成
- **lint/format**: 自動修正を優先、手動対応が必要な場合のみ報告
- **レビュー指摘**: パターンを蓄積し、次回以降の自動チェックに反映

## TDD ワークフロー

```text
1. テストを書く（Red）
2. 最小限の実装でテストを通す（Green）
3. リファクタリング（Refactor）
4. テストが通ることを確認
```

- 振る舞いの変更には必ずテストを追加・修正する
- テストが通る状態を維持してからリファクタする
