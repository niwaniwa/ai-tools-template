# ドキュメントライフサイクル管理方式の選定

## コンテキスト

プロジェクトのドキュメント（ADR・仕様書・調査）にライフサイクル（wip/accepted/archive）を導入したい。
AIツールが accepted なドキュメントのみを実装の根拠として参照できるようにする必要がある。

## 検討した選択肢

### 選択肢 A: ディレクトリベース（wip/, accepted/, archive/）

- 利点: ディレクトリを見るだけでステータスがわかる
- 欠点: ステータス変更時にファイル移動が必要。パスが変わるためリンク切れが発生する

### 選択肢 B: ファイル名プレフィックス（wip--xxx.md）

- 利点: grepしやすい
- 欠点: ファイル名が長くなる。リネーム時にgit履歴が途切れる

### 選択肢 C: YAML frontmatter の status フィールド

- 利点: パスが安定する。CIやAIツールでパース可能。ADRツールの標準的な手法
- 欠点: ファイルを開かないとステータスがわからない

## 決定

**選択肢 C: YAML frontmatter** を採用する。

パスの安定性がAIツール連携とリンク管理に最も重要であり、
ステータスの視認性はCIのバリデーションで補完できるため。

## 影響

- 正: ドキュメントのパスが変わらないため、相互参照が安定する
- 正: CI でfrontmatterを検証し、ステータスの整合性を自動チェックできる
- 負: エクスプローラーでステータスを一覧するには追加ツールが必要
- リスク: frontmatter を書き忘れた場合、CIで検出する仕組みが必要
